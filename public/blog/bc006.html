Dynamic Report Sorting: Synchronizing Page Views with Reports

Background

In Business Central, users often perform heavy data analysis directly on List Pages. They apply complex filters and specific sort orders (e.g., sorting by a custom FlowField or a specific Posting Date). However, when they run a standard report, those UI-level customizations are lost, forcing them to re-apply filters on the Report Request Page—which often doesn't even support the same sorting flexibility.

The Problem Standard Report Request Pages are frequently limited to sorting by Primary Keys or specific keys defined in the Report Dataset. This creates a "manual gap" where the user's on-screen analysis cannot be easily exported into a professional report format.

Requirement

The goal is to create a seamless "Print what I see" experience. If a user has filtered a list to show "Overdue Invoices" and sorted them by "Remaining Amount," the report should inherit that exact state with zero manual intervention.

Solution Overview

Instead of hardcoding sorting logic into the report or creating multiple report versions, we use a bridge pattern. We capture the View of the current Page and "inject" it into the Report's data engine before it fires.

  • Inherit Filters: Carry over all Quick Filters and Filter Pane criteria.
  • Inherit Sorting: Respect the column-click sorting active in the Web Client.

Technical Implementation

Record.GetView Record.SetView Report.SetTableView

The General Implementation Pattern

This logic can be placed in any Page or Page Extension action. Replace YourRecord and YourReport with your specific objects.In this case I use Customer List and Report Customer Sales Statistics.

CustomerListExt.al
action(ALECustomerSales)
{
    Image = Sales;
    Caption = 'ALE Sales Statistics';
    ApplicationArea = all;
    ToolTip = 'Executes the ALE Sales Statistics action.';
    trigger OnAction()
    var
        customer: Record Customer;
        CustomerSalesStatistics: Report "Customer Sales Statistics";
    begin
        customer.SetView(Rec.GetView());
        CustomerSalesStatistics.SetTableView(customer);
        CustomerSalesStatistics.Run();
    end;
            }

Pros

  • Maximize Standard Features: No need to create custom versions of complex standard reports.
  • Minimal Customization: Reduces the extension footprint, making future BC upgrades much easier.
  • Intuitive UI: Users get exactly what they expect based on their current screen view.
