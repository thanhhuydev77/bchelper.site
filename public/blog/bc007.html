Mastering RDLC: Creating a Smart 2-Column Report for Warehouse Labels

Background

In warehouse operations, efficiency is everything. Managers generate reports linking Container Nos. to their respective Warehouse Shipments, typically printed on 4x6 inch paper—the logistics standard. These labels are critical for staff to execute loading tasks accurately.

The Problem: Vertical Overflow A single container can hold up to 10 shipments. Using a standard single-column RDLC layout on a small 4x6 label quickly leads to space issues. The list becomes too long, forcing unnecessary page breaks or cutting off vital information, which wastes paper and confuses the staff.

New Solution: Dynamic 2-Column Split

To maximize the 4x6 real estate, we implement a dual-column layout. When shipments exceed a defined limit (e.g., 5 rows), the list automatically "overflows" into a second column on the same page. This keeps all data on a single label.

Technical Implementation

RDLC SSRS AL Dictionary Warehouse Logistics

Step 1: Generate a Dynamic Index in AL

To split the data in RDLC, we need to "rank" each shipment within its container. Using an AL Dictionary ensures $O(1)$ performance for tracking and incrementing indexes on the fly.

ALELoadWarehouseShpmtNew.al
report 70004 "ALE Load Whse Shipment New"
{
    Caption = 'Load Warehouse Shipment New';
    UsageCategory = Administration;
    ApplicationArea = All;
    DefaultRenderingLayout = Layout;

    dataset
    {
        dataitem("Warehouse Shipment"; "Warehouse Shipment Header")
        {
            column(No_; "No.") { }
            column(External_Document_No_; "External Document No.") { }
            column(ExtDocNoBarCode; EncodeBarcode128("External Document No.")) { }
            column(Index; GetIndexByExternalDocumentNo("External Document No.", "No.")) { }
        }
    }
    rendering
    {
        layout(Layout)
        {
            Type = RDLC;
            LayoutFile = '.\Layout\ALELoadWarehouseShipment2.rdlc';
        }
    }
    trigger OnInitReport()
    begin
        gBarcodeFontProvider := Enum::"Barcode Font Provider"::IDAutomation1D;
    end;

    local procedure EncodeBarcode128(gBarcodeText: Text): Text
    begin
        if gBarcodeText <> '' then
            exit(gBarcodeFontProvider.EncodeFont(gBarcodeText, "Barcode Symbology"::Code128));
    end;

    procedure GetIndexByExternalDocumentNo(RootKey: Code[35]; Value: code[20]): Integer
    begin
        if not gContainterWSH.Add(RootKey, 1) then
            gContainterWSH.Set(RootKey, gContainterWSH.Get(RootKey) + 1);

        exit(gContainterWSH.Get(RootKey));
    end;

    var
        gBarcodeFontProvider: Interface "Barcode Font Provider";
        gContainterWSH: Dictionary of [Code[35], Integer];
}

Step 2: Structure the RDLC Layout

Instead of a standard Row Group (vertical growth), we use two identical table objects placed horizontally within a main container cell.

Step 3: Side-by-Side Filtering

Since RDLC doesn't support "newspaper-style" columns natively, we apply filters to the two child tables:

  • Left Table: Set Filter property to Index <= 5.
  • Right Table: Set Filter property to Index > 5.

If a container has only 4 shipments, the right table remains empty, keeping the label clean and professional.

Pros and Cons

Space Efficiency: Maximizes the 4x6 label width, doubling the data capacity.

Better UX: Staff see the entire loading list at a glance without flipping through multiple labels.

High Performance: The Dictionary approach in AL ensures fast execution even with large datasets.
